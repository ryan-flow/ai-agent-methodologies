# 约束优先排查法（Constraint-First Debugging）

## 核心原则

**先验证不可控层是否支持，再决定是否在可控层投入精力。**

很多功能看起来是"代码写得不对"，实际是"平台根本不支持"。搞反了就是浪费时间。

## 排查层级（自底向上）

1. **平台/API 层**（不可控）— 官方文档、API reference、已知限制
2. **协议/格式层**（半可控）— 数据格式、编码、传输协议
3. **代码/逻辑层**（可控）— 我们能改的一切

## 执行步骤

1. **明确目标**：我们要实现什么具体效果？
2. **查文档**：目标平台的官方 API 文档是否明确支持这个能力？
3. **找证据**：搜索社区、GitHub issues、Stack Overflow，确认有没有人成功过
4. **快速验证**：用最简单的方式（curl / 一行代码）直接测试 API 是否接受这个请求
5. **如果 API 层不支持 → 停下来**，不要试图通过改客户端代码绕过去
6. **如果 API 层支持 → 再往代码层排查**

## 检查清单

动手前过一遍：

- [ ] 官方文档里有没有明确说支持这个功能？
- [ ] 有没有人（GitHub issue / 论坛）成功实现过？
- [ ] 最简单的 API 调用测试做了吗？
- [ ] 如果 API 不支持，有没有替代方案能达到 80% 效果？

## 反面案例

### 微信语音条发送（2026.05.26）

**目标**：让 Hermes 机器人发出微信原生绿色语音气泡

**错误路径**：
1. 改 `send_voice()` 代码 → 2. 装 pilk 编码器 → 3. 改正则表达式 → 4. 改文件扩展名识别 → 一路改下去

**最终发现**：iLink Bot API 的 `sendmessage` 不支持 `voice_item`（`ret=-2`），是腾讯服务端拒绝

**浪费**：2-3 小时的代码修改全部无用

**正确做法**：第一步就该用 curl 直接调 API 发个 voice_item，看返回值

### 对比：微信语音接收（同期）

- 先确认 API 是否支持接收语音 → 支持
- 再确认格式 → SILK/AMR
- 然后才开始写转换代码
- ✅ 流程正确

## 适用场景

- 第三方 API 集成
- 平台功能开发（微信、Telegram、Discord 等）
- 任何涉及"平台是否支持 X"的判断
- 调试时感觉改了很多代码但问题依旧
